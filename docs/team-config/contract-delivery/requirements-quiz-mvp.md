> **文档身份**：产品侦探需求挖掘报告（Contract for Delivery · 执行契约）  
> **版本**：v1.0  
> **出具日期**：2026-06-18

## 〇、执行摘要（TL;DR）

一句话定义本 MVP：**让榔头已有的 UGC 笔记在 6 周内具备"被玩"的能力，而不是仅"被看"的能力。**

- 这不是"做一个答题游戏"，而是"给平台加一个让内容变现的玩法形态"。
- 这不是"抢用户时长"，而是"把本会 3 天就沉没的笔记，延长到 30 天生命周期"。
- 这不是"先做大 DAU 数字"，而是"先找到 500 位真愿意为 UGC 关卡付费的种子用户 + 50 位愿意持续出关卡的创作者"。

## 1. 核心目标 —— 这个 MVP 最终解决谁的什么问题？

### 1.1 被服务的「三类人」（必须同时服务，缺一视为失败）

| 角色 | 痛点 | MVP 必须交付的"解药" | 谁在代码里对应哪个服务 |
|------|------|----------------------|------------------------|
| **C 端玩家（16–28 岁女性，学生/职场新人）** | 刷短视频焦虑、想学点东西但坐不住、想玩游戏但怕被说"不务正业" | 打开榔头 → 看到一篇笔记 → 一键切成 10 道选择题 → 30 秒完成一局 → 获得"学到了"的自我确认 + 排行榜爽感 | Content Service（笔记源）+ AI Service（生成题）+ Game Service（Session/Leaderboard）+ Interact Service（分享） |
| **中腰部创作者（1 万–100 万粉，美妆/知识/情感/职场垂类）** | 广告单价走低、笔记发完即沉、粉丝无法沉淀为"主动关系" | 写完一篇笔记 → 勾选"允许 AI 生成关卡" → 系统自动出 10 道题 → 看到"被玩次数 / 粉丝闯关完成率 / 分成金额" | Creator Service（Wallet）+ AI Service + Content Service |
| **榔头平台** | UGC 内容 ARPU = 0、用户停留时长短、没有差异化留存钩子 | 把单篇笔记 ARPU 从 0 元 → 15 元区间，把次留从 X% → X+10% | Ad Service（广告）+ Game Service（Inventory/Payment） |

### 1.2 可被量化的核心命题

- **命题 P1（玩家侧）**：7 天内，至少 30% 的首次使用答题玩法的用户，会回来玩第二局。
- **命题 P2（创作者侧）**：前 6 周内，至少 50 位创作者主动勾选"笔记开放为关卡"。
- **命题 P3（平台侧）**：单篇开放为关卡的笔记，其"二次消费时长"是未开放笔记的 5 倍以上。

> **不可模糊**：三个命题必须全部成立，才算 MVP 成功。任一命题为假即触发熔断。

## 2. 边界定义 —— 明确不做哪些事？哪些用户角色被排除？

### 2.1 本 MVP **绝对不做**的事（Out of Scope）

| # | 不做 | 原因 |
|---|------|------|
| O1 | 多人实时对战答题 | 开发成本 × 3、依赖 Game Session 重构，留给第二阶段 |
| O2 | 创作者手工编辑题库（AI 出完题后只能"一键采纳 / 一键重生成"，不能逐题修改） | 手工编辑器是方向 B（剧情模组）的能力，MVP 不需要 |
| O3 | 非 UGC 来源的题库（不做官方题库、不做 IP 授权题） | 偏离"UGC 即关卡"的核心主张 |
| O4 | 卡牌养成 / 角色系统 / 抽卡 | 合规风险高 + 开发周期长，属于方向 C |
| O5 | 关卡内电商带货（笔记挂商品链的下单抽佣） | 依赖完整电商履约，不适合 MVP |
| O6 | 跨 App 分享链路之外的"社交炫耀墙"（仅做排行榜 + 单条分享到微信/QQ/微博） | 多平台深度打通成本大 |
| O7 | Android 原生之外的独立答题 App | 必须嵌在现有榔头 App / H5 内部 |
| O8 | 题目音频、动画、图片素材生成 | 题干仅支持纯文本 + 原笔记中的 1 张封面图 |
| O9 | 题目数量 > 12 题 / < 5 题 | MVP 固定 10 题，便于 AI Prompt 收敛 |
| O10 | 付费续命 > 2 次 / 局 | 避免被投诉"玩不起"，也防止沉迷 |

### 2.2 明确排除的用户角色

| # | 被排除的角色 | 排除理由 |
|---|--------------|----------|
| X1 | 16 岁以下未成年人 | 合规红线，未成年人模式强制屏蔽答题入口 |
| X2 | 专业"答题职业玩家"（刷榜党） | Game Service 反作弊逻辑将其识别并限流，不做付费外挂 |
| X3 | 完全没有笔记产出能力的"纯看客"用户 | 仍可玩，但不作为深度运营对象 |
| X4 | 粉丝 > 500 万的头部 KOL | 其内容版权/商业合作复杂，留给第二阶段签约 |
| X5 | 非中文内容创作者（小语种、少数民族语言） | MVP 仅支持简体中文题库生成 |
| X6 | 企业号 / 品牌号 | 其内容多为营销性质，AI 生成题目质量差，且不参与创作者分成 |

### 2.3 边界的"灰色地带"处理原则

- **原则 1（UGC 纯净性）**：任何情况下，不引入"非 UGC 笔记"作为题目源。
- **原则 2（AI 主导）**：任何情况下，不提供"人工出题后台"给运营团队。若 AI 质量不够，宁可暂时降低题库量，也不退回人工模式。
- **原则 3（单 App 闭环）**：任何情况下，答题玩法必须在榔头 App 内完成闭环，不得跳转到第三方 H5 独立站。

## 3. 禁区清单 —— 哪些数据绝对不能碰？哪些操作需要强制审批？

### 3.1 数据禁区（Red Line Data）

| # | 数据 | 禁区等级 | 禁止原因 |
|---|------|----------|----------|
| R1 | 用户手机号、真实姓名、身份证号 | 🔴 绝对禁区 | 隐私合规，GDPR + 《个人信息保护法》 |
| R2 | 用户定位精确到街道/门牌号 | 🔴 绝对禁区 | 同上 |
| R3 | 未成年人身份标识（TeenMode 下的用户 ID 列表） | 🔴 绝对禁区 | 青少年保护 |
| R4 | 创作者钱包余额 / 银行卡 / 提现账户 | 🔴 绝对禁区 | 金融合规 |
| R5 | 用户站内私信内容（Message Service 的 message 表） | 🔴 绝对禁区 | 通讯秘密 |
| R6 | 答题历史中用户的"错题记录"用于算法推荐时的反向识别 | 🟡 强制脱敏 | 仅允许聚合统计（如"错题分布"），不允许对单用户画像反向识别 |
| R7 | 笔记中出现的第三方品牌 logo、受版权保护的图片 | 🟡 强制审查 | 版权风险 |
| R8 | AI 生成题库的原始 Prompt 内容（含用户笔记原文） | 🟡 内部受控 | 仅 AI Service 内部可见，禁止日志打印 |

### 3.2 操作审批清单（Require Sign-Off）

| # | 操作 | 必须审批人 | 审批流程 |
|---|------|------------|----------|
| A1 | 上线前关闭答题入口 | CEO + CTO | 双人审批邮件留痕 |
| A2 | 调整付费点金额（续命币 0.99 元 / 关卡门票） | CEO + 财务负责人 | 工单 + 合同 |
| A3 | 启用"题目人工审核"回退策略（即放弃纯 AI 生成） | CEO + 产品总监 | 必须附数据报告证明 AI 质量不达标 |
| A4 | 向任何第三方（AI 厂商、广告商）导出用户答题行为数据 | CTO + 法务 | 签署数据处理协议 |
| A5 | 在微信以外的社交渠道新增分享链路 | 产品总监 | 评估 DAU 贡献 |
| A6 | 把答题玩法接入未成年人模式（即解除 X1 排除） | CEO + 法务 + 青少年保护官 | 三人联签 |
| A7 | 更改"UGC 笔记 → 关卡"的授权方式（从"勾选授权"改为"默认授权"） | CEO + 法务 | 需 80% 以上创作者同意调研 |
| A8 | 题目数量从 10 题改为其他值 | 产品总监 + QA 负责人 | 需重新验证加载时长假设 |

### 3.3 日志与审计强制要求

- 所有 A 类操作必须写入 `audit_log` 表（Content Service 已有 `AuditLog` 实体，扩展 `operation_type = 'QUIZ_APPROVAL'`）。
- AI 生成题库的每次调用必须记录 `note_id → prompt_hash → question_set_id → latency_ms`，供 AI 厂商对账。
- 付费行为必须双写 Game Service + Ad Service + Creator Service 的日志，防止单边账。

## 4. 验收标准 —— 3 个"没实现就算失败"的场景

> **测试原则**：以下 3 个场景必须全部端到端打通，任何一个失败 = MVP 不可验收。

### 场景 S1：笔记一键变关卡（核心主链路）

```
前置条件：
  - 已登录创作者账号（user.role = CREATOR）
  - 创作者有一篇已发布的公开笔记（note.status = PUBLISHED, note.quiz_enabled = 1）
  - AI Service 正常可用

操作步骤：
  1. 创作者在笔记详情页点击"生成闯关题"
  2. 系统调用 POST /api/v1/ai/quiz/generate { note_id } （走 AiCreationController 扩展端点）
  3. AI Service 在 30 秒内返回 10 道选择题（题干 20–60 字，4 选项，1 正确答案）
  4. Content Service 将题目集写入 quiz_set 表，关联 note_id
  5. 创作者在笔记页看到"已生成关卡"徽章

验收通过条件：
  ✅ 95 分位响应时间 ≤ 30 秒
  ✅ 生成题目数 = 10
  ✅ 每题包含题干 + 4 选项 + 正确答案 + 解析
  ✅ 数据库 quiz_set 表有记录，note.quiz_enabled 自动置为 1
  ✅ 创作者无需修改即可发布
```

### 场景 S2：玩家一局闯关 + 续命 + 排行榜

```
前置条件：
  - 已登录普通用户（非未成年）
  - 有至少 1 个已发布的 quiz_set
  - 用户钱包有 ≥ 0.99 元余额（或已绑定支付）

操作步骤：
  1. 用户在首页 Feed 刷到一篇带"可闯关"徽章的笔记
  2. 点击进入关卡，Game Session 创建（POST /api/v1/game/quiz/start）
  3. 系统出 10 题，倒计时 60 秒/题，每题生命值 1
  4. 答对 7 题以上 → 通关，获得"学霸"标签
  5. 生命值归零 → 弹出"续命 0.99 元"选项（最多 2 次）
  6. 完成后，成绩写入 GameLeaderboard，好友关系链可见排名
  7. 支持一键分享到微信好友

验收通过条件：
  ✅ 关卡加载 ≤ 2 秒（首屏）
  ✅ 续命支付链路 10 秒内完成（走 GamePayment + 微信支付）
  ✅ 排行榜数据 30 秒内同步到好友侧
  ✅ 分享卡片包含：关卡名称 + 用户得分 + 排名 + 立即挑战按钮
  ✅ 未成年人账号无法进入此链路
```

### 场景 S3：创作者分成结算

```
前置条件：
  - 有至少 1 个创作者的关卡被 100+ 玩家通关
  - Ad Service 已记录对应题库的广告展示 1000+ 次
  - CreatorWallet 表有该创作者账户

操作步骤：
  1. 每周一凌晨 02:00，结算定时任务触发
  2. 计算：分成 = 广告收入 × 30% + 付费续命收入 × 20%（按 note_id 归因）
  3. 写入 CreatorWallet.weekly_revenue
  4. 创作者后台（CreatorService）看到"本周闯关分成 X 元"
  5. 创作者点击"提现" → 走现有 WalletService → T+1 到账

验收通过条件：
  ✅ 归因准确率 100%（玩家/广告行为必须精确关联到 note_id + quiz_set_id）
  ✅ 结算任务幂等（重复执行结果一致）
  ✅ 创作者后台数字与财务对账一致（误差 ≤ 0.01 元）
  ✅ 提现走现有 WalletService，无需新建支付链路
```

### 4.1 验收的"一票否决"条款

- **V1**：任何一次测试中出现"AI 生成题目正确答案 ≥ 2 个"（即答案本身错误），视为 P0 级缺陷；连续 3 次出现即封锁发布。
- **V2**：未成年人账号通过任何路径进入答题玩法，视为合规事故。
- **V3**：付费续命成功但 GameSession 未更新分数，视为资金事故，必须全额退款。

## 5. 失败后果 —— 最坏情况是什么？降级方案？

### 5.1 三条"坏到极点"的失败场景

| # | 最坏情况 | 触发条件 | 损害等级 |
|---|----------|----------|----------|
| F1 | **AI 生成题目质量崩盘**：题库瞎出、答案错误、被用户在社交平台骂"AI 缝合怪"，导致品牌声誉受损 + 大量差评 | 上线 4 周内"愿意再玩一次比例" < 20% | 🔴 致命 |
| F2 | **DAU 冷启动失败**：答题入口日活 < 1000，且分享率 < 5%，无法形成数据飞轮 | 上线 8 周 DAU < 1 万 | 🔴 致命 |
| F3 | **付费转化失败**：广告主不买单、用户不充钱、创作者因分成低不接入 | 上线 12 周付费转化率 < 2% + 广告主投放金额 < 10 万/月 | 🔴 致命 |

### 5.2 分级降级方案（Contingency Plan）

#### 降级等级 L1：功能降级（保留核心命题）

- **触发**：AI 题库正确率 ≥ 70% 但 < 85%
- **方案**：
  1. 暂时关闭"创作者一键开关卡"入口，仅运营精选 Top 100 笔记开放为关卡
  2. AI 每生成一批题（100 题），运营人工抽检 5 题，错 1 题打回
  3. 排行榜只展示"正确率"而非"得分"，弱化 AI 错误题的影响

#### 降级等级 L2：模型降级（保留"UGC → 关卡"主链路）

- **触发**：AI 厂商接口宕机 > 24 小时 / 调用成本超支 3 倍
- **方案**：
  1. 切换为"模板化题库"：针对 5 类垂类（美妆/知识/情感/职场/母婴）预置 50 套模板，AI 仅做"填空"而非"自由生成"
  2. 调用成本下降 60%，质量稳定性提升
  3. 暂时关闭非 5 类垂类笔记的关卡生成入口

#### 降级等级 L3：商业模型降级（保留用户侧体验）

- **触发**：付费转化率 < 2% 且广告主 ROI < 1.2
- **方案**：
  1. 关闭"续命币"付费点，改为"看广告续 3 次命"（Ad Service 激励视频）
  2. 创作者分成改为"每通关 1 次 = 0.05 元保底"，不再与广告挂钩
  3. 所有关卡免费开放，收入 100% 来自广告

#### 降级等级 L4：战略转型（保住公司）

- **触发**：CEO 熔断条款——18 周内三条止损线同时被击穿
- **方案**：
  1. 立即停止 C 端答题玩法的对外推广
  2. 把 AI 题库生成能力、Game Session、Leaderboard 打包为 SDK
  3. 面向 B 端中小内容平台 / MCN 输出"UGC 游戏化 SaaS"服务
  4. 项目组 80% 资源转向 B 端销售 + 客户成功

### 5.3 降级启动权（谁有权按下"降级按钮"）

| 降级等级 | 启动权 | 审批 |
|----------|--------|------|
| L1 | 产品总监 | CEO 事后知会 |
| L2 | CTO | CEO 事前批准 |
| L3 | CEO 本人 | 董事会事后知会 |
| L4 | CEO 本人 + 董事会联签 | 董事会决议 |

## 6. 成功证据 —— 上线后哪 3 个指标改善能证明做对了？

> **指标选择原则**：
> 1. 必须能在现有 9 微服务的监控体系内直接度量（无外部数据采集成本）
> 2. 必须是"对比基线"指标，而非绝对值
> 3. 必须能同时覆盖玩家侧、创作者侧、平台侧三个视角

### 指标 K1：UGC 内容二次消费率（玩家侧 · 核心）

| 维度 | 说明 |
|------|------|
| **定义** | （通过答题玩法进入笔记的 UV × 平均停留时长）÷ 笔记总 UV，对比基线（未开放答题时）的比值 |
| **数据源** | Interact Service 的 ShareRecord + Content Service 的 NoteFeedVO + Game Service 的 GameSession |
| **采集点** | Prometheus 指标 `lt_quiz_content_secondary_rate`，1 分钟粒度，写入 Loki |
| **基线** | 上线前 2 周的"笔记阅读 → 点赞/收藏"转化率（预期 ≈ 15%） |
| **成功阈值** | 上线 4 周后 ≥ 30%（即二次消费率翻倍） |
| **CEO 评语** | "这是 MVP 最核心的北极星指标——证明'玩笔记'比'看笔记'更有价值" |

### 指标 K2：创作者周分成 Top 10 集中度（创作者侧 · 生态）

| 维度 | 说明 |
|------|------|
| **定义** | Top 10 创作者的周闯关分成 ÷ 全部创作者周闯关分成 |
| **数据源** | Creator Service 的 WalletService + Ad Service 的 AdImpression |
| **采集点** | 每周一生成报表，写入 CreatorDashboardVO |
| **基线** | 上线前 0（因为没有闯关分成） |
| **成功阈值** | 上线 8 周后 Top 10 集中度 ≤ 40%（即至少有 30+ 创作者同时在赚钱） |
| **CEO 评语** | "如果 Top 10 集中度 > 60%，说明我们养了 10 个头部但没养出生态，必须调整分成规则给中腰部创作者更多倾斜" |

### 指标 K3：答题玩法次留 vs 平台整体次留差值（平台侧 · 留存）

| 维度 | 说明 |
|------|------|
| **定义** | （玩过答题玩法的用户次留 − 未玩过答题玩法的用户次留）× 100% |
| **数据源** | User Service 的 UserProfile + Game Service 的 GameSession + Interact Service 的 ShareRecord |
| **采集点** | UserLevelService 每日聚合，写入 Prometheus `lt_quiz_retention_delta` |
| **基线** | 上线前 0（两个群体无差异） |
| **成功阈值** | 上线 6 周后 ≥ 5 个百分点（即玩答题的用户比不玩的用户次留高 5%） |
| **CEO 评语** | "这证明答题玩法不是'抢走用户时长'，而是'吸引并留住更优质的用户'" |

### 6.1 三个指标的"正反馈闭环"

```
K1（内容二次消费率）↑  →  更多创作者愿意接入  →  K2（创作者集中度）健康
        ↓                                                      ↓
  更多新玩家加入  ←──────────────────────────────────  更多新玩家看到好题
        ↓
K3（留存差值）↑  →  平台整体 DAU 上涨  →  广告主 ROI 达标  →  更多预算投入
```

> 三个指标必须**同步改善**才算真成功。如果 K1 涨但 K3 不涨，说明玩法成了"一次性热点"；如果 K2 集中度过高，说明生态不健康。

### 6.2 指标的"熔断阈值"

| 指标 | 熔断阈值（8 周仍未达到） | 触发动作 |
|------|--------------------------|----------|
| K1 | < 20%（对比基线仅微升） | 触发 L1 降级，重做 AI 出题 Prompt |
| K2 | Top 10 集中度 > 70% | 调整分成规则，给中腰部创作者加权 |
| K3 | < 0%（玩答题的用户次留反而更低） | 立即暂停答题入口 2 周，做用户访谈 |
