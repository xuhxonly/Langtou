## 二、数据库表命名规范

### 2.1 表命名规则

- **全部小写 + 下划线分隔**（snake_case）
- **单数形式**：`user`、`note`、`like_record`（禁止 `users`、`notes`）
- 表名前**不加模块前缀**（与服务解耦）：`quiz_set`、`quiz_question` 而非 `t_quiz_set`
- 关联表使用 `{主体}_{客体}`：`note_tag`、`user_level`、`note_product`
- 冗余/历史表以 `_legacy` 结尾标记

### 2.3 字段命名规范

- 主键：统一 `id` BIGINT UNSIGNED AUTO_INCREMENT
- 外键：`{关联表}_id`（如 `user_id`、`quiz_set_id`）
- 时间：`created_at` / `updated_at` DATETIME(3) 毫秒精度
- 软删除：`deleted` TINYINT(1) 默认 0（0 正常 / 1 已删除）
- 状态：`status` VARCHAR(32) 或 TINYINT，枚举值见"四、状态枚举定义"
- 排序：`sort_order` INT
- JSON：统一使用 MySQL JSON 类型，命名为 `{名词}_json`（如 `images_json`）
- 版本号：乐观锁字段统一为 `version` INT
- 扩展：预留 `ext_json` 字段存非结构化扩展数据

## 三、通用 DTO 结构约定

### 3.1 顶层响应包装（所有 API 统一）

```java
// com.langtou.common.result.Result<T>
{
  "code": 200,          // Integer, 业务状态码（见"六、错误码定义"）
  "message": "操作成功", // String, 人类可读消息（支持 i18n）
  "data": { ... },      // T, 业务数据；批量响应时为数组
  "timestamp": 1700000000000  // Long, 服务器时间戳（毫秒）
}
```

### 3.2 分页结构

```java
// com.langtou.common.result.PageResult<T>
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "list": [ ... ],           // List<T>, 当前页数据
    "pageNo": 1,               // int, 从 1 开始
    "pageSize": 20,            // int
    "total": 1234,             // long
    "totalPages": 62           // int, 可选
  },
  "timestamp": 1700000000000
}
```

### 3.3 DTO 命名规范

| 后缀 | 用途 | 示例 |
|------|------|------|
| `DTO`（Data Transfer Object） | **入参**（接收客户端请求） | `UserRegisterDTO`、`QuizGenerateRequest` |
| `VO`（View Object） | **出参**（返回给客户端的视图对象） | `UserProfileVO`、`NoteFeedVO`、`GameInventoryVO` |
| `Request` | 复杂入参，强调"动作" | `QuizSubmitRequest`、`AiDraftRequest` |
| `Response` | 复杂出参，强调"结果" | `QuizGenerateResponse`、`AiCoverRecommendResponse` |
| `Entity`（MyBatis-Plus `@TableName`） | 数据库映射对象 | `User`、`QuizSet` |

**入参类禁止使用 `DTO` 与 `Request` 混合命名**——要么用 DTO、要么用 Request，全项目二选一（历史代码存在混用，新增代码**优先使用 DTO** 作为入参后缀，Response 保持不变）。

### 3.4 通用字段约定

所有跨服务传输的 DTO/VO **必须**包含：
- `id` Long：主键
- `createdAt` / `updatedAt` String（ISO-8601，毫秒精度）
- 可选 `ext` Map<String, Object>：扩展字段，避免频繁加列

### 3.5 AI / Quiz 专用 DTO 约束

- `AiDraftRequest` 中 `prompt` 字段必须经过长度校验（≤ 2000 字）
- `QuizGenerateRequest` 中 `difficulty` 取值为 `EASY` / `MEDIUM` / `HARD`
- `QuizSubmitRequest` 必须带 `attemptId`（客户端首次发起答题时获取）

### 4.3 通用枚举模式建议

- 所有枚举类实现 `IEnum` 接口，提供 `getValue()` 与 `getDesc()`

## 五、API 路由规范

### 5.1 路由结构

```
/api/v{version}/{service}/{resource}[/{identifier}][/{action}]
```

| 段 | 说明 | 示例 |
|----|------|------|
| `/api` | 固定前缀 | — |
| `/v1` | 版本号 | 升级时才改，路径保持向后兼容 |
| `/{service}` | 服务名（对应端口 808x） | `/user`、`/content`、`/quiz` |
| `/{resource}` | 资源名（复数，kebab-case） | `/notes`、`/quiz-sets` |
| `/{identifier}` | 资源标识（可选） | `/123`、`/by-slug/summer-2026` |
| `/{action}` | 自定义动作（可选） | `/publish`、`/submit` |

### 5.2 路由示例

| 方法 | 完整路径 | 说明 |
|------|---------|------|
| POST | `/api/v1/quiz/sets` | 创建答题关卡 |
| GET | `/api/v1/quiz/sets/{id}` | 获取关卡详情 |
| POST | `/api/v1/quiz/sets/{id}/start` | 开始答题 |
| POST | `/api/v1/quiz/attempts/{id}/submit` | 提交答题 |
| GET | `/api/v1/user/profile` | 获取当前用户资料 |
| POST | `/api/v1/auth/login/sms` | 短信登录 |
| GET | `/api/v1/content/notes/feed` | 首页信息流 |
| POST | `/api/v1/interact/likes/{targetId}` | 点赞 |
| DELETE | `/api/v1/interact/likes/{targetId}` | 取消点赞 |
| GET | `/api/v1/game/leaderboard/season/{seasonId}` | 赛季排行榜 |

### 5.3 版本演进规则

- **当前版本**：`v1`
- 破坏性变更时才升级版本号（路径、参数、响应结构任一不兼容）
- 新版本发布后，旧版本至少维护 **6 个月**
- 废弃版本需在响应头 `Sunset` 字段注明最后可用日期

### 6.4 错误码使用规则

- **一个语义一个码**：禁止多个场景复用同一错误码
- **码段独占**：每个服务只在分配的区段内分配新码
- **i18n 同步**：新增错误码必须在 `langtou-common/src/main/resources/i18n/messages_zh_CN.properties` 与 `messages_en_US.properties` 中同步增加翻译
- **废弃码**：禁止重用已下线的错误码，只增不改

## 七、Redis Key 命名约定

```
lt:{module}:{sub_module}:{identifier}
```

| 模块 | 示例 | 说明 |
|------|------|------|
| 用户 | `lt:user:profile:{userId}` | 用户资料缓存 |
| 内容 | `lt:content:note:{noteId}` | 笔记详情缓存 |
| 互动 | `lt:interact:like:{userId}:{noteId}` | 是否已点赞 |
| 游戏 | `lt:game:session:{sessionId}` | 对局会话状态 |
| 答题 | `lt:quiz:attempt:{attemptId}` | 答题进行中状态（带 TTL） |
| 答题 | `lt:quiz:rank:{seasonId}` | 排行榜 ZSet |
| 限流 | `lt:ratelimit:{ip}:{api}` | 网关限流计数器 |
| 验证码 | `lt:auth:sms:{phone}` | 短信验证码（TTL 5min） |

**通用规则**：
- TTL 必须明确；登录态默认 7 天，验证码默认 5 分钟
- 禁止在 Key 中出现特殊字符（空格、`#`、`/`）
- `identifier` 优先使用业务主键，避免使用复合字符串
