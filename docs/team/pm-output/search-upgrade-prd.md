## 1. 背景与目标

### 1.1 背景

当前榔头平台的搜索功能仅支持基础的关键词匹配，存在以下问题：

- **搜索效率低**: 用户需要输入完整关键词才能获得有效结果，缺乏联想和补全能力
- **发现能力弱**: 缺少热搜榜等引导机制，用户不知道"搜什么"
- **结果质量差**: 搜索结果无排序优化，无筛选能力，关键词不高亮，用户难以快速定位目标内容
- **搜索历史缺失**: 用户无法回顾和管理搜索记录

搜索是内容社区的核心入口之一，搜索体验的优劣直接影响内容分发效率和用户留存。本次升级旨在全面提升搜索体验，使其成为用户发现内容的重要渠道。

### 1.2 目标

| 目标类型 | 具体目标 | 衡量指标 |
|----------|----------|----------|
| **搜索效率** | 提升搜索命中率和速度 | 搜索结果点击率从15%提升至30% |
| **搜索活跃** | 提升搜索功能使用率 | 搜索渗透率（搜索用户/DAU）从20%提升至40% |
| **内容发现** | 增强用户内容发现能力 | 搜索到内容的用户次日留存率提升10% |
| **搜索满意度** | 提升用户搜索满意度 | 搜索无结果率从25%降低至10% |

### 1.3 范围

本文档涵盖搜索体验升级的三大模块：

1. **搜索建议/自动补全** - 输入时的实时联想建议
2. **热搜榜** - 平台热门搜索词展示
3. **搜索结果优化** - 多维度筛选、关键词高亮、搜索历史管理

## 2. 搜索建议/自动补全

### 2.1 功能概述

用户在搜索框输入时，实时展示联想建议词，帮助用户快速找到目标内容，减少输入成本，提升搜索效率。

### 2.2 用户故事

| 编号 | 用户故事 | 验收标准 |
|------|----------|----------|
| SUG-US-01 | 作为用户，我希望在输入关键词时系统能实时展示联想词，以便快速找到想搜的内容 | 输入首字符后200ms内展示建议列表 |
| SUG-US-02 | 作为用户，我希望建议列表能区分历史搜索和热门联想，以便快速选择或发现新内容 | 建议列表分类展示（历史/热门/联想），类别标识清晰 |
| SUG-US-03 | 作为用户，我希望点击建议词后直接跳转搜索结果页，以便快速查看结果 | 点击建议词后直接执行搜索并展示结果 |
| SUG-US-04 | 作为用户，我希望空输入状态下能看到我的历史搜索和热门搜索，以便快速复用或发现热点 | 聚焦搜索框但未输入时，展示历史搜索和热门搜索 |
| SUG-US-05 | 作为用户，我希望可以删除单条历史搜索记录，也可以一键清空，以便管理隐私 | 支持左滑删除单条记录，支持一键清空全部历史 |

#### 2.3.3 建议词排序算法

**联想建议排序公式**:

```
Score = w1 * PrefixMatchScore + w2 * SearchFrequencyScore + w3 * ContentCountScore + w4 * RecencyScore
```

| 因子 | 说明 | 权重 |
|------|------|------|
| PrefixMatchScore | 前缀匹配度（完整匹配 > 前缀匹配 > 包含匹配） | 0.35 |
| SearchFrequencyScore | 该词被搜索的频次（标准化） | 0.30 |
| ContentCountScore | 该词对应的内容数量（标准化） | 0.20 |
| RecencyScore | 该词近期的搜索热度趋势 | 0.15 |

### 2.4 技术方案

#### 2.4.1 架构设计

```
客户端输入
  -> 防抖(200ms)
  -> GET /api/v1/search/suggestions?q={keyword}&limit=8
  -> 搜索建议服务
    -> Redis ZRANGEBYLEX 前缀匹配 (联想词库)
    -> Redis ZREVRANGE 历史搜索 (用户维度)
    -> Redis ZREVRANGE 热门搜索 (全局维度)
  -> 聚合排序 -> 返回建议列表
```

#### 2.4.2 数据结构

**Redis 数据结构设计**:

| Key | 类型 | 说明 | 示例 |
|-----|------|------|------|
| `search:suggestions:all` | Sorted Set | 全局联想词库 | score=搜索频次, member=搜索词 |
| `search:history:{userId}` | Sorted Set | 用户搜索历史 | score=时间戳, member=搜索词 |
| `search:hot:daily` | Sorted Set | 每日热搜榜 | score=搜索频次, member=搜索词 |
| `search:hot:weekly` | Sorted Set | 周热搜榜 | score=搜索频次, member=搜索词 |

**联想词库更新策略**:

| 数据来源 | 更新频率 | 更新方式 |
|----------|----------|----------|
| 用户搜索词 | 实时 | 用户执行搜索后，异步写入 Sorted Set，INCRBY 更新频次 |
| 内容标题/标签 | 日更新 | 每日凌晨从内容库提取新词，写入联想词库 |
| 运营配置词 | 实时 | 运营后台手动添加/编辑热搜词 |
| 低频词清理 | 周清理 | 每周清理频次<5且超过30天未搜索的词 |

#### 2.4.3 接口设计

**搜索建议接口**:

```
GET /api/v1/search/suggestions

请求参数:
  - q (string, 可选): 输入关键词，为空时返回历史+热门
  - limit (int, 可选): 返回数量，默认8，最大20
  - type (string, 可选): all/history/hot/suggest，默认all

响应示例 (q为空时):
{

  "data": {
    "history": [

    "hot": [
      {"keyword": "夏季穿搭", "rank": 1, "heat": 98500, "trend": "up"},
      {"keyword": "防晒推荐", "rank": 2, "heat": 87200, "trend": "stable"}
    ]
  }
}

响应示例 (q="美"时):
{
  "code": 0,
  "data": {
    "suggestions": [
      {"keyword": "美妆教程", "type": "history", "highlight": "<b>美</b>妆教程"},
      {"keyword": "美食推荐", "type": "suggest", "highlight": "<b>美</b>食推荐"},
      {"keyword": "美白护肤", "type": "suggest", "highlight": "<b>美</b>白护肤"}

```

### 3.1 功能概述

在搜索入口和搜索结果页展示平台当前热门搜索词，帮助用户快速了解平台热点，引导内容发现。

### 3.2 用户故事

| 编号 | 用户故事 | 验收标准 |

| HOT-US-02 | 作为用户，我希望热搜词有热度标识和趋势变化，以便判断话题热度 | 每条热搜显示热度值和升降趋势 |

| 数据项 | 说明 |

### 3.4 技术方案

#### 3.4.1 热搜统计流程

```
用户搜索行为
  -> Kafka 搜索日志 Topic
  -> Flink 实时流处理
    -> 关键词提取与标准化
    -> 敏感词过滤

    -> Redis ZINCRBY 实时计数

    -> 计算排名与趋势
    -> 写入热搜榜缓存
```

## 4. 搜索结果优化

### 4.1 功能概述

优化搜索结果页的展示和交互能力，包括多维度筛选、关键词高亮和搜索历史管理，帮助用户快速定位目标内容。

### 4.2 用户故事

| 编号 | 用户故事 | 验收标准 |
|------|----------|----------|
| RES-US-01 | 作为用户，我希望搜索结果能按内容类型筛选，以便只看某一类型的内容 | 支持按笔记/视频/用户/话题筛选 |
| RES-US-02 | 作为用户，我希望搜索结果能按时间范围筛选，以便查看最新或特定时期的内容 | 支持全部/一天内/一周内/一月内/一年内 |
| RES-US-03 | 作为用户，我希望搜索结果能按不同维度排序，以便快速找到最相关或最热门的内容 | 支持综合排序/最新发布/最多点赞/最热评论 |
| RES-US-US-04 | 作为用户，我希望搜索结果中的关键词能高亮显示，以便快速判断内容相关性 | 搜索关键词在标题和摘要中高亮标记 |
| RES-US-05 | 作为用户，我希望可以管理搜索历史，以便保护隐私或快速复用 | 支持查看/删除单条/清空全部搜索历史 |

#### 4.3.3 综合排序算法

```
综合排序分数 = w1 * TextRelevance + w2 * ContentQuality + w3 * Popularity + w4 * Recency + w5 * Personalization
```

| 因子 | 说明 | 权重 |
|------|------|------|
| TextRelevance | 关键词与内容的文本相关性（BM25） | 0.30 |
| ContentQuality | 内容质量分（基于互动率/完播率/审核评分） | 0.25 |
| Popularity | 内容热度（点赞数/评论数/收藏数标准化） | 0.20 |

### 4.5 搜索历史管理

#### 4.5.1 功能设计

| 功能 | 说明 | 交互方式 |
|------|------|----------|
| **查看历史** | 展示近30天搜索记录，按时间倒序 | 搜索框聚焦时展示 |
| **删除单条** | 删除指定搜索记录 | 左滑展示删除按钮 |
| **清空全部** | 清空所有搜索历史 | 点击"清空"按钮，二次确认 |
| **搜索频次** | 同一关键词多次搜索，显示搜索次数 | 历史记录右侧显示次数 |
| **跨端同步** | 搜索历史在多设备间同步 | 后端存储，登录用户自动同步 |

#### 4.5.2 搜索历史存储方案

| 存储位置 | 数据结构 | 说明 |
|----------|----------|------|
| Redis | `search:history:{userId}` (Sorted Set) | 热数据缓存，score=时间戳 |
| MySQL | `user_search_history` 表 | 持久化存储，保留30天 |

**数据表设计**:

```sql
CREATE TABLE user_search_history (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    keyword VARCHAR(100) NOT NULL,
    search_count INT DEFAULT 1,
    last_search_at DATETIME NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_time (user_id, last_search_at DESC),
    INDEX idx_user_keyword (user_id, keyword)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 4.5.3 接口设计

**搜索历史接口**:

## 7. 数据指标

### 7.1 核心指标

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|----------|--------|
| 搜索渗透率 | 使用搜索功能的用户占比 | 搜索用户数 / DAU | > 40% |
| 搜索结果点击率 | 用户点击搜索结果的比例 | 搜索结果点击数 / 搜索结果曝光数 | > 30% |
| 搜索无结果率 | 搜索无结果的搜索占比 | 无结果搜索次数 / 总搜索次数 | < 10% |
| 建议采纳率 | 用户点击建议词的比例 | 建议点击数 / 建议展示数 | > 25% |
| 搜索到内容次日留存 | 通过搜索发现内容的用户次日留存 | 搜索用户次日回访数 / 搜索用户数 | > 45% |

### 7.2 辅助指标

| 指标 | 定义 | 用途 |
|------|------|------|
| 人均搜索次数 | 平均每用户每天搜索次数 | 评估搜索活跃度 |
| 搜索结果零点击率 | 搜索后未点击任何结果的比例 | 评估搜索质量 |
| 筛选功能使用率 | 使用筛选功能的搜索占比 | 评估筛选需求 |
| 热搜点击率 | 热搜词被点击的比例 | 评估热搜引导效果 |
| 搜索历史回用率 | 点击历史搜索的比例 | 评估历史功能价值 |
