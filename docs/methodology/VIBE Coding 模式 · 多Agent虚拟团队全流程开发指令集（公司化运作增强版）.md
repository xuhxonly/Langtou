# VIBE Coding 模式 · 多Agent虚拟团队全流程开发指令集（公司化运作增强版）

> 本版本以原版十五大阶段为基础，深度融合公司化运作视角，新增美术设计、策略设计、故事剧情设计、市场营销、运营增长、用户获取等全链路角色与流程，覆盖从市场洞察到商业成功的完整闭环。

## 一、项目启动与战略规划阶段

### 1.1 项目发起与方向定义

**指令：** 用户提供项目创意、题材方向或市场机会点后，AI需执行以下初始化动作：

1. **机会识别**：分析用户输入，判断该项目属于何种品类（游戏/工具/SaaS/内容平台等），并输出初步的《机会洞察报告》，包含市场规模、目标用户画像、核心价值主张。

2. **战略定位**：定义产品在市场中的定位（差异化/成本领先/细分市场专注），明确项目的商业目标（赚取收益/获取用户/品牌建设/技术沉淀等）。

3. **可行性初判**：基于技术复杂度、资源需求、时间窗口，给出项目可行性评分和风险预警。

### 1.2 项目初始化配置（系统级）

**指令：** AI自动生成完整的项目初始化结构：

- **文件结构模板**：按照分层架构（表现层/业务层/数据层/基础设施层）生成项目目录骨架。

- **依赖管理**：根据技术选型生成 `package.json`/`requirements.txt`/`go.mod`/`Cargo.toml` 等，并锁定核心依赖版本。

- **配置文件体系**：生成 `.env.example`（环境变量模板）、`config.yaml`（业务配置）、`appsettings.json`（应用配置）、`log-config.xml`（日志配置）。

- **代码规范工具**：配置 ESLint/Prettier（前端）、Black/isort（Python）、golangci-lint（Go）、Checkstyle（Java）、clang-format（C++），并生成 `.editorconfig` 统一编辑器风格。

- **版本控制初始化**：执行 `git init`，生成 `.gitignore`（包含OS文件、IDE配置、依赖目录、构建产物、敏感信息等），创建初始 `README.md` 和 `CHANGELOG.md`，完成首次提交。

- **开发环境标准化**：生成 `devcontainer.json`（容器化开发环境）或 `setup.sh` 一键环境配置脚本，确保团队成员开发环境一致。

### 1.3 Skill技能体系初始化

**指令：** 建立项目级Skill技能库索引：

- **通用技能**：代码生成（多语言）、代码审查、单元测试生成、文档生成、重构建议、性能分析、安全审计。

- **专业领域技能**：Unity/Unreal引擎开发、React/Vue前端开发、Spring Boot/Node.js后端开发、数据库设计与优化、DevOps流水线构建、AI模型集成与调优。

- **业务技能**：需求分析、竞品调研、用户画像构建、商业模式设计、增长黑客策略。

- **技能安装与更新机制**：定义每个Skill的触发条件、执行流程、输出格式和版本管理策略，确保技能可追溯、可回滚。

### 2.2 团队协作与治理机制

**指令：** 建立以下协作规范：

- **汇报关系矩阵**：定义每个角色的直接汇报对象和虚线汇报对象，明确决策路径。

- **信息同步机制**：
  
  - 每日站会（15分钟）：各成员汇报昨日进展、今日计划、阻塞问题。
  
  - 每周迭代评审：展示已完成功能，收集反馈，调整优先级。
  
  - 每月战略回顾：复盘关键指标，评估里程碑达成情况，调整战略方向。

- **决策权限划分**：
  
  - 战略级决策（如产品方向变更、大额投入）：CEO + 核心领导团队。
  
  - 战术级决策（如功能优先级、技术选型）：相关角色自主决策，事后同步。
  
  - 执行级决策：各成员自主执行，按规范记录。

- **介入与退出条件**：
  
  - 各角色在产品生命周期各阶段（需求/设计/开发/测试/发布/运营）的介入时机与退出条件明确定义。
  
  - 关键阶段门禁（Gate）：需求评审门禁、设计评审门禁、代码审查门禁、测试通过门禁、发布审批门禁。

| 美术评审会 | 按需   | 美术总监、策划、PM   | 美术资源确认、风格统一检查        |
| 市场周会  | 每周   | 市场、运营、产品     | 市场数据复盘、策略调整          |

## 三、市场调研与产品规划阶段

### 3.1 市场机会识别与竞品分析

**指令：** 由市场经理主导，产品经理协同，完成以下工作：

1. **市场宏观分析（PESTEL）**：
   
   - 政治因素（政策监管、数据安全法、版号政策等）
   
   - 经济因素（目标市场消费水平、付费意愿）
   
   - 社会因素（文化偏好、用户习惯、人口结构）
   
   - 技术因素（新兴技术可用性、平台演进）
   
   - 环境因素（可持续发展要求）
   
   - 法律因素（知识产权、隐私合规）

2. **竞品对标分析（深度维度）**：
   
   - **核心玩法对比**：机制差异、创新点、操作手感
   
   - **用户体验对比**：新手指引、交互流畅度、UI/UX品质
   
   - **变现模式对比**：付费点设计、定价策略、VIP体系、Battle Pass
   
   - **技术架构对比**：引擎版本、网络同步方案、数据架构
   
   - **社区运营对比**：社群活跃度、UGC生态、KOL合作模式
   
   - **更新频率与内容节奏**：版本周期、内容量、活动密度
   
   - **用户评价情感分析**：App Store/Google Play/Steam评论爬取与NLP情感评分
   
   - **用户画像对比**：年龄、性别、地域、付费习惯

3. **SWOT分析报告**：
   
   - 内部优势（技术积累、团队能力、IP资源）
   
   - 内部劣势（资源限制、经验缺口、品牌认知度）
   
   - 外部机会（市场空白、技术变革、用户需求变化）
   
   - 外部威胁（强竞品、政策风险、替代品）

4. **竞争定位图**：以两个关键维度（如"玩法深度 vs 上手门槛"、"付费深度 vs 社交属性"）绘制四象限定位图，明确产品位置。

### 3.2 用户需求与洞察

**指令：** 由用户研究员主导，完成以下工作：

1. **用户画像构建**：基于数据（或假设验证）建立3-5个典型用户画像，包含年龄、职业、游戏习惯、付费意愿、痛点、爽点。

2. **用户旅程地图**：绘制用户从"认知→考虑→下载→首次打开→新手引导→核心体验→留存/付费→推荐/流失"的完整旅程，标注各环节情绪曲线。

3. **需求挖掘方法**：深度访谈、焦点小组、问卷调查（需设计样本量和置信度）、数据分析（已有产品时）。

### 3.3 产品需求文档（PRD）输出

**指令：** 由产品经理主导撰写PRD，包含以下章节：

1. **产品概述**：产品愿景、目标市场、核心价值主张、差异化卖点。

2. **用户故事与用例**：按用户角色编写的功能需求用例（如"作为玩家，我希望..."）。

3. **功能需求列表**（按模块划分）：
   
   - 核心功能（P0 - MVP必须包含）
   
   - 增强功能（P1 - 版本1.x包含）
   
   - 锦上添花功能（P2 - 后续版本）

4. **非功能需求**：性能指标（响应时间/并发数）、安全要求、可用性要求、兼容性要求。

5. **约束与假设**：技术约束、资源约束、外部依赖、假设条件。

6. **成功标准（KPI）**：可量化的指标，如次留>40%、日活>10万、付费率>5%等。

### 3.4 需求评审与优先级排序

**指令：** 由PM组织需求评审会议，参与角色：系统架构师、开发专家、测试工程师、市场经理、运营经理。

- **可行性分析**：每一项需求的技术可行性、资源估算、风险等级。

- **MoSCoW优先级排序**：
  
  - **Must have**（必须有）：没有则产品不可用
  
  - **Should have**（应该有）：重要但可灰度
  
  - **Could have**（可以有）：锦上添花
  
  - **Won't have**（这次不做）：明确排除，管理预期

- **MVP范围定义**：明确第一个可发布版本的最小功能集。

- **迭代路线图**：规划后续3-6个迭代的功能交付计划。

### 3.5 项目立项与计划制定

**指令：** 由项目经理主导，CEO审批，完成正式立项：

1. **项目章程**：项目名称、背景、目标、范围、主要干系人、项目经理/负责人。

2. **项目计划书**：
   
   - 里程碑节点（需求冻结→设计完成→开发完成→测试通过→发布上线）
   
   - 时间估算（各阶段工时估算，含缓冲）
   
   - 资源分配（人力/预算/基础设施）
   
   - 验收标准（各阶段门禁条件）
   
   - 沟通计划（会议安排、报告机制）

3. **风险评估与应急预案**：
   
   - 技术风险（新技术不确定性、集成风险）→ 预案：技术预研Spike、备选方案
   
   - 人员风险（关键人员离职、技能缺口）→ 预案：知识沉淀、Backup培养
   
   - 时间风险（延期风险）→ 预案：范围缩减、增加资源
   
   - 市场风险（需求变化、竞争加剧）→ 预案：敏捷响应、定期市场扫描
   
   - 合规风险（政策变化）→ 预案：法务提前介入、合规检查清单
   
   - 预算风险（超支）→ 预案：预算缓冲、ROI定期复审

4. **项目看板初始化**：在Trello/Jira/Notion中创建项目面板，设定列（需求池→待开发→开发中→待测试→测试中→已完成→已发布），配置自动化规则。

## 四、系统架构与设计阶段

### 4.1 系统架构设计

**指令：** 由系统架构师主导输出《系统架构设计文档》：

1. **架构视图（4+1视图）** ：
   
   - 逻辑视图：模块划分、分层架构、关键类/服务
   
   - 开发视图：包结构、工程组织、依赖关系
   
   - 进程视图：运行时进程/线程模型、并发设计
   
   - 物理视图：部署拓扑、网络架构、硬件配置
   
   - 场景视图：关键流程时序图/序列图

2. **技术栈选型与理由**：
   
   - 前端：框架选择（React/Vue/Flutter/Unity）、状态管理方案
   
   - 后端：语言（Go/Java/Python/Node）、框架（Spring Boot/Gin/Django）、RPC方案
   
   - 数据层：关系型数据库（PostgreSQL/MySQL）、NoSQL（Redis/MongoDB）、搜索引擎
   
   - 中间件：消息队列（Kafka/RabbitMQ）、缓存、分布式事务方案
   
   - 基础设施：云服务商、容器编排、CI/CD工具

3. **接口定义（OpenAPI/GraphQL/gRPC）**：
   
   - 内外接口清单、请求/响应格式、错误码体系、版本管理策略

4. **数据架构**：
   
   - ER图、数据字典、分库分表策略、数据生命周期管理、归档策略

5. **部署拓扑**：
   
   - 开发/测试/预发布/生产环境规划、高可用设计、灾备方案

6. **可扩展性设计**：
   
   - 插件机制、模块热插拔、配置化/低代码化设计、开放API规划

7. **高可用与SLA目标**：
   
   - 可用性目标（99.9%）、RTO/RPO定义、降级/熔断/限流策略

### 4.2 编码规范与开发公约

**指令：** 由软件开发专家制定《编码规范手册》：

1. **命名规范**：各语言命名风格（驼峰/蛇形/帕斯卡）、禁止缩写清单、领域术语表。

2. **注释与文档规范**：文件头注释、函数注释（含参数/返回值/异常）、复杂逻辑行内注释、CHANGELOG维护规则。

3. **异常处理策略**：异常类型定义、捕获粒度、日志记录级别、用户友好提示。

4. **日志记录规范**：日志级别定义（DEBUG/INFO/WARN/ERROR/FATAL）、日志格式、敏感信息脱敏规则。

5. **安全编码要求**：输入校验、SQL注入/XSS/CSRF防护、加密存储、密钥管理规范。

6. **代码示例与模板**：各场景的标准代码模板（Controller/Service/Repository/工具类）。

7. **代码评审清单**：提交前自检清单、评审关注点、阻塞性问题定义。

### 4.3 美术风格与规范制定

**指令：** 由美术总监主导，输出《美术设计规范手册》：

1. **风格定义**：整体美术风格方向（写实/卡通/像素/水墨等）、参考作品、色彩体系（主色/辅助色/强调色）、光影氛围。

2. **角色设计规范**：比例标准、服装风格、表情规范、体型变化范围、三视图要求。

3. **场景设计规范**：构图原则、远景/中景/近景层次、环境光色温、氛围参考。

4. **UI美术规范**：界面统一风格、按钮状态（默认/悬停/点击/禁用）、图标风格与尺寸规范、字体与排版。

5. **特效规范**：特效色彩限制、粒子数量上限、特效时长标准、性能预算。

6. **资源命名与目录规范**：文件命名规则、目录结构、版本管理方式。

7. **分辨率与性能标准**：各平台目标分辨率、三角面数限制、Draw Call预算、纹理压缩格式。

### 4.4 叙事与世界观设计

**指令：** 由世界观架构师与叙事设计师主导，输出《世界观设定集》与《叙事设计文档》：

1. **世界观设定**：世界背景、历史脉络、地理/势力/种族设定、核心冲突与主题、关键事件时间线。

2. **角色设定**：主要角色档案（背景/性格/外貌/能力/成长弧光）、NPC设计原则、角色关系图谱。

3. **主线剧情**：三幕式结构、关键剧情节点、结局设计。

4. **支线任务体系**：支线类型（角色支线/世界探索/收集/挑战）、与主线的交织方式。

5. **对话与文案风格**：语气统一标准、各角色说话风格、关键台词设计原则。

6. **叙事节奏与玩法整合**：剧情触发时机、演出形式（对话/过场/环境叙事）、玩法与叙事的融合点。

## 五、代码开发与迭代交付阶段

### 5.1 敏捷迭代开发流程

**指令：** 建立标准化的敏捷开发流程：

1. **迭代周期**：固定周期（建议1-2周），每个迭代有明确的目标和可交付成果。

2. **迭代规划**：
   
   - 迭代开始前，PM与团队进行需求梳理和任务拆分（User Story → Task）
   
   - 团队估算工作量（Story Point或工时），承诺迭代容量
   
   - 明确每个任务的验收标准（DoD - Definition of Done）

3. **每日站会**：各成员汇报昨日进展、今日计划、阻塞问题，PM/Scrum Master负责跟进阻塞消除。

4. **并行开发与协作**：
   
   - 前后端/客户端按照接口契约并行开发
   
   - 使用Mock数据和契约测试（Pact）确保接口兼容
   
   - 每日代码合并，解决冲突，确保主干持续可构建

5. **代码审查**：
   
   - 所有提交需经过至少一位审查者批准（除紧急修复外）
   
   - 审查关注点：逻辑正确性、代码可读性、性能隐患、安全漏洞、测试覆盖
   
   - 审查工具：GitHub PR / GitLab MR / Gerrit

6. **持续集成**：
   
   - 每次提交触发自动化构建、单元测试、代码静态检查（SonarQube）
   
   - 构建失败或测试失败需立即修复，主干始终保持绿色

### 5.2 代码重构与健康度管理

**指令：** 定期执行代码健康度检查：

1. **重构触发条件**：代码坏味（重复代码/长函数/大类/过深嵌套）、性能瓶颈、安全漏洞、技术债累积超阈值。

2. **重构计划**：列入迭代Backlog，每次重构必须有明确的产出目标和验证标准。

3. **性能剖析（Profiling）**：
   
   - 客户端：帧率检测、内存占用、启动耗时、包体大小
   
   - 服务端：响应时间、吞吐量、CPU/内存使用率、数据库慢查询
   
   - 使用工具：Unity Profiler/Chrome DevTools/JProfiler/pprof/JMeter

4. **技术债管理**：建立技术债登记表，按严重程度分级，定期评审并分配资源偿还。

### 5.3 游戏/产品特色开发专项

**指令：** 针对游戏类产品，需关注以下特色模块的开发：

1. **新手引导系统**：
   
   - 渐进式引导（非强迫）、智能触发（基于玩家行为）
   
   - 可跳过、可回看、可查阅（帮助中心）
   
   - 引导与游戏进程的自然融合

2. **随机事件与动态难度系统**：
   
   - 随机事件池设计、触发概率动态调整、玩家行为影响事件走向
   
   - 动态难度匹配（基于玩家表现实时调整挑战）
   
   - AI大模型集成：实时生成个性化事件或对话

3. **成长与成就体系**：
   
   - 等级/技能/装备/收集等多维度成长
   
   - 成就系统（里程碑/挑战/隐藏成就）
   
   - 成就分享与社会化展示

4. **社交系统**：
   
   - 好友系统、实时对战/组队、公会/战队
   
   - 社交排行榜、好友赠送/互助
   
   - 观战/ replay分享

5. **活动与运营系统**：
   
   - 限时活动配置化（无需发版即可上线新活动）
   
   - Battle Pass / 赛季制
   
   - 签到/任务/节日活动模板化

6. **个性化与AI增强**：
   
   - 基于大模型的NPC对话（千人千面）
   
   - AI辅助策略提示（如"如虎添翼"功能：智能推荐装备/策略）
   
   - 玩家行为分析与个性化内容推荐
   
   - 自适应难度（让每个玩家都在"心流通道"中）

### 5.4 SDK/平台化提炼与复用

**指令：** 在开发过程中同步思考平台化能力：

1. **可复用模块识别**：引擎核心、资源管理、用户系统、支付系统、数据分析、社交模块等。

2. **SDK抽象设计**：
   
   - 标准API接口定义、配置化开关、插件机制
   
   - 提供默认实现 + 允许定制扩展
   
   - 版本管理与向后兼容策略

3. **文档与示例**：SDK使用文档、快速开始指南、完整示例项目、FAQ。

4. **Low-Code/No-Code底座**：
   
   - 场景编辑器、事件触发器、数据驱动配置
   
   - 资源热更新机制、AB测试配置平台
   
   - 运营活动配置后台（非技术人员可用）

### 5.5 平台适配与兼容性开发

**指令：** 确认并实施多平台适配策略：

1. **目标平台确认**：Web、移动端（iOS/Android）、PC（Windows/Mac/Linux）、主机（PS/Xbox/Switch）、小程序（微信/抖音）等。

2. **适配策略**：
   
   - **UI自适应**：响应式布局（Flex/Grid）、锚点系统、安全区域适配
   
   - **分辨率适配**：多档分辨率设计、资源分级加载（低/中/高配）
   
   - **输入适配**：键鼠/触控/手柄的输入映射与交互优化
   
   - **网络适配**：离线模式、弱网优化（重试/降级/预加载）
   
   - **性能适配**：低端设备降级方案（LOD、特效简化、分辨率缩放）

3. **半自动化适配方案**：
   
   - 配置中心动态调整UI布局（不同设备返回不同布局配置）
   
   - 资源按需加载（根据设备性能自动选择资源等级）
   
   - 平台特定代码隔离（条件编译/平台判断）

4. **兼容性测试矩阵**：列出各平台/设备/OS版本的测试覆盖范围，手工+自动化结合。

## 六、测试与质量保障阶段

### 6.1 测试策略与计划

**指令：** 由QA经理与测试工程师制定《测试策略文档》：

1. **测试金字塔**：
   
   - 单元测试（占比70%+）：覆盖核心业务逻辑、工具类、数据层
   
   - 集成测试（占比20%）：接口测试、模块间集成、数据库集成
   
   - 端到端测试（占比10%）：核心用户旅程、关键业务流程
   
   - 契约测试：前后端接口契约验证

2. **测试类型与覆盖**：
   
   - **功能测试**：需求覆盖100%，含正常/异常/边界用例
   
   - **性能测试**：响应时间、并发用户、吞吐量
   
   - **压力测试**：极限负载、系统崩溃点、自动恢复验证
   
   - **兼容性测试**：各平台/设备/OS版本/分辨率/网络环境
   
   - **安全测试**：OWASP Top 10、渗透测试、数据泄露检测
   
   - **可用性测试**：真实用户操作、任务完成率、满意度评分
   
   - **回归测试**：自动化回归套件、冒烟测试用例集
   
   - **探索性测试**：基于风险的深度探索、寻找隐藏缺陷

3. **测试环境管理**：
   
   - 开发环境（DEV）、集成环境（INT）、测试环境（TEST）、预发布环境（STAGING）、生产环境（PROD）

### 6.2 测试用例与自动化

**指令：** 测试工程师执行：

   - 移动端：Appium/XCUITest/Espresso
   
   - 接口层：Postman/Newman/RestAssured/pytest
   
   - 性能层：JMeter/LoadRunner/k6
   
   - 单元测试：JUnit/pytest/Jest/Mocha + 覆盖率工具

3. **持续测试**：CI流水线集成自动化测试，每次提交运行冒烟测试套件，每日运行完整回归套件。

### 6.3 QA质量保障

**指令：** QA工程师独立执行质量审查：

1. **QA独立性原则**：QA团队独立于开发团队，直接向CTO或质量委员会汇报质量状态。

2. **QA审查范围**：
   
   - 需求阶段：需求可测试性评审、测试要点识别
   
   - 设计阶段：设计可测试性评审、测试策略初步制定
   
   - 编码阶段：代码质量抽查、单元测试覆盖率审查
   
   - 测试阶段：测试过程监督、测试结果审计、缺陷趋势分析
   
   - 发布阶段：质量门禁把关、发布风险评估、发布后质量监控

3. **缺陷管理流程**：
   
   - 缺陷报告格式（环境/复现步骤/期望/实际/日志/截图）
   
   - 缺陷等级定义（Blocker/Critical/Major/Minor/Trivial）
   
   - 缺陷生命周期（New→Open→Fixed→Verified→Closed / Reopen）
   
   - 缺陷评审会议（周会）、缺陷趋势分析、缺陷预防措施

4. **质量度量与报告**：
   
   - 缺陷密度、缺陷逃逸率、测试覆盖率、自动化通过率
   
   - 发布阻塞问题数量、平均修复时间（MTTR）
   
   - 每周质量报告、迭代质量回顾

### 6.4 质量门禁标准（Quality Gate）

**指令：** 定义各阶段的质量准入与准出条件：

| 阶段   | 准入门槛          | 准出条件                                   |
| ---- | ------------- | -------------------------------------- |
| 需求阶段 | 市场调研完成、战略方向确认 | PRD完成、需求评审通过、优先级定案                     |
| 设计阶段 | PRD已冻结        | 架构设计文档完成、技术评审通过、设计稿评审通过                |
| 编码阶段 | 设计文档通过评审      | 单元测试覆盖率>80%、代码评审通过、静态扫描无高危问题           |
| 测试阶段 | 代码提交至测试环境     | 测试用例通过率>95%、无Blocker/Critical缺陷、性能指标达标 |
| 发布阶段 | 测试阶段通过、所有门禁通过 | UAT通过、安全审计通过、发布审批签署、回滚预案就绪             |
