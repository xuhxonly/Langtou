# VIBE Coding 模式 · 多Agent虚拟团队开发全流程提示词指令集（完整版）

## 一、项目启动与初始化阶段

### 1.1 通用方法与流程定义

请提供一套基于 VIBE Coding 模式进行外部编程的通用方法和完整操作步骤。例如，第一步由用户指定项目的开发方向（如题材、类型、核心玩法等）；第二步，AI 在接到指令后，应首先进行项目建立的系统初始化配置，包括生成初始化的程序代码、设定开发限制与约束条件、构建系统级的提示词模板等，并明确需要调用的相关 Skill 技能及其安装要求，以及是否需要进一步搜集外部资料或进行技术调研。

### 1.2 初始化配置细化

在初始化配置阶段，AI 应自动生成项目文件结构模板、依赖管理文件（如 `package.json`、`requirements.txt`）、基础配置文件（如 `.env.example`、`config.yaml`），并设定代码规范（如 ESLint、Prettier、Black 等），同时初始化版本控制仓库（如 Git）并生成初始提交记录。

## 二、虚拟开发团队构建与角色配置

### 2.1 团队构建

构建一个多 Agent 角色的虚拟开发团队，团队成员应至少包括产品经理、系统架构师、项目经理、软件开发专家、各模块的开发工程师及测试工程师等核心岗位。

### 2.2 团队协作机制

在构建虚拟团队时，应明确每位 Agent 的汇报关系、协作接口、信息同步机制（如每日站会、迭代评审），并设定各角色在需求、设计、开发、测试、部署各阶段的介入时机与退出条件。

### 2.3 成员技能与提示词配置

为上述多 Agent 虚拟团队中的每一位成员配置相应的 Skill 技能和系统提示词模板，明确各自的职责分工，以提升各成员的专业性，并清晰界定其工作内容和协作边界。

### 2.4 技能分类与版本管理

Skill 技能的配置应区分通用技能（如代码生成、代码审查、测试用例生成）与专业技能（如 Unity/Unreal 开发、数据库设计、UI/UX 设计），并为每位 Agent 设定系统提示词模板的版本管理机制，确保可追溯、可回滚。

### 2.5 团队完整性确认

请问目前是否已构建完整的虚拟开发团队？团队中是否包含产品经理、系统架构师、软件开发专家、项目经理、后端开发工程师、前端开发工程师、系统开发工程师、测试工程师、美术师、剧情设计师等全部必要角色？

## 三、需求分析与产品规划阶段

### 3.1 市场竞品分析与需求规划

由产品经理开展市场竞品分析，完成市场调研与需求规划，输出产品需求文档；同时，系统架构师负责提出产品的概要设计、系统架构方案及技术选型。

### 3.2 竞品对标详细要求

竞品对标应包含：核心玩法对比、用户体验对比、变现模式对比、技术架构对比、社区运营对比、更新频率对比及用户评价情感分析，输出 SWOT 分析报告和竞争定位图。

### 3.3 需求评审

请产品经理组织需求评审会议，邀请系统架构师、开发专家、测试工程师共同参与，对需求文档中的每一项功能进行可行性分析和优先级排序（MoSCoW 法则），明确 MVP 范围与后续迭代计划。

### 3.4 项目立项与计划制定

由项目经理正式立项，并组织团队进入后续的开发、迭代与交付阶段。请补全后续所有开发流程与协作细节，并对上述步骤进行重新梳理，确保整体流程的完整性与可执行性。

### 3.5 立项细化

立项后应输出项目计划书，包含里程碑节点、风险评估、资源分配、时间估算及验收标准。后续开发应遵循敏捷迭代流程，每轮迭代结束后进行回顾与复盘。

### 3.6 风险评估

请项目经理组织风险识别会议，列出技术风险、人员风险、时间风险、市场风险及合规风险，制定应急预案与缓解措施，并在每个迭代中跟踪风险状态变化。

## 四、系统架构与设计阶段

### 4.1 架构设计文档输出

请系统架构师输出系统架构设计文档，包含模块划分图、数据流图、接口定义、数据库 ER 图、部署拓扑图及技术栈详细说明，并组织技术评审，确保架构的可扩展性与高可用性。

### 4.2 编码规范制定

请软件开发专家制定编码规范手册，包含命名规则、注释标准、异常处理策略、日志记录规范及安全编码要求，并通过代码示例进行宣贯，确保团队编码风格一致。

## 五、代码开发与迭代阶段

### 5.1 代码变动与产品衍生

请对当前的程序代码进行适当的变动与重构，以此为基础衍生出另一款具有差异化的游戏产品。

### 5.2 衍生品差异化要求

衍生新游戏时，应明确差异化维度（如玩法机制、美术风格、目标用户、付费模式），并评估技术复用率与改造工作量，确保衍生品与原型产品形成互补而非同质竞争。

### 5.3 用户友好功能拓展

请继续对代码和功能进行优化完善，强化用户友好的交互设计，拓展更多实用功能，提升整体体验。用户友好功能拓展应包括：新手引导系统、操作反馈优化、无障碍设计（如色盲模式、字体缩放）、多语言国际化支持、以及社交分享与成就系统等。

### 5.4 并行开发与接口兼容

请前后端及系统开发工程师按照既定接口文档进行并行开发，并使用 Mock 数据与契约测试（Pact）确保接口兼容性；每日进行代码合并，解决冲突，保持主干稳定。

### 5.5 趣味性增强

目前两款游戏产品的趣味性略显不足，请问如何增强其趣味性和精彩程度，以更有效地吸引玩家？趣味性增强可考虑加入：随机事件系统、成长养成体系、竞技排行机制、隐藏彩蛋、社交互动（好友对战/组队）、限时活动及剧情分支选择等模块。

### 5.6 美术与剧情团队补充

当前项目缺少音效、背景音乐，且美术图片精致度不足。因此，建议在团队中新增美术师和剧情策划师角色，进一步对游戏的视听表现和叙事内容进行完善与优化。

### 5.7 美术与剧情职责明确

美术师应负责角色设计、场景原画、UI 界面、特效动画、图标 Logo 等全套视觉体系；剧情策划师应撰写世界观设定、角色背景、主线支线任务剧本及对话文案，并与玩法设计紧密结合。

### 5.8 团队自主扩展机制

团队可根据项目实际需要，自主决定是否新增 Agent 角色（例如美术师），并为其配置合适的 Skill 技能和系统提示词模板。请全体团队成员进行头脑风暴，共同确定产品开发的下一步工作内容。请所有成员高效协作，立即执行上述工作，同时尝试在现有代码基础上提炼出一套可复用的游戏开发平台，以便未来基于该平台快速定制开发多款游戏产品。

### 5.9 游戏开发平台提炼

游戏开发平台的提炼应包含：通用游戏引擎核心、资源管理模块、场景编辑器、事件触发器、数据持久化方案、用户系统、支付接口及分析统计插件，形成可配置、可扩展的 Low-Code/No-Code 底座。

### 5.10 平台可扩展性确认

请问能否将当前游戏程序代码抽象封装为一套 SDK 平台，以便未来在该平台上快速构建和发布多种不同类型的游戏应用产品？SDK 平台应提供标准 API 文档、示例项目、调试工具、版本更新日志、兼容性说明及开发者社区支持，并设计插件机制以便第三方开发者扩展功能。

### 5.11 代码结构优化

请对程序代码进行系统化、结构化、规范化、模块化的整理与提升，同时优化和完善虚拟团队中多 Agent 角色的工作内容、职责分工、Skill 技能配置及提示词模板设计。

### 5.12 代码重构与性能剖析

请团队成员定期进行代码重构与性能剖析（Profiling），识别性能瓶颈与代码坏味，制定重构计划，确保代码长期健康可维护。

## 六、测试与质量保障阶段

### 6.1 首次全面测试

请测试工程师对程序的所有功能模块进行全面的诊断和系统性测试，输出测试报告。

### 6.2 首次问题修复

请开发团队根据测试反馈，及时修复所有已发现的问题。

### 6.3 修复后回归测试

请测试工程师再次对程序各项功能进行全面诊断与测试，确保修复后的稳定性。

### 6.4 测试计划与用例编写

请测试工程师编写测试计划与测试用例文档，覆盖正常流程、异常流程、边界条件及极端场景，并使用自动化测试框架（如 Selenium、JUnit、pytest）提升回归效率。

### 6.5 QA 工程师配置与职责

请为团队配置高效的 QA 工程师，设置合适的 Skill 技能和系统提示词模板，明确其职责范围。要求 QA 工程师对整个 SDK 和应用产品进行系统性的 QA 测试，并就发现的所有质量问题和缺陷，推动开发团队及时跟进解决。

### 6.6 QA 与测试协同

请 QA 工程师与测试工程师协同配合，对 SDK Platform 和游戏 APP 进行全面问题诊断与质量检测，深入发掘潜在的隐性问题，并将问题清单提交开发团队，以便及时修复，提升系统整体稳定性。

### 6.7 测试类别与标准

QA 测试应覆盖：功能测试、性能测试、压力测试、兼容性测试、安全测试、可用性测试及回归测试，每个测试类别应设定通过标准与严重等级划分（如 Blocker/Critical/Major/Minor/Trivial）。

### 6.8 综合测试与修复闭环

请测试工程师对本项目程序代码进行全面测试，发现问题并记录；开发团队需及时跟进测试结果，修复所有问题；QA 工程师则需对 SDK 和游戏产品进行独立的 QA 质量审查，确保整体质量达标。

### 6.9 质量门禁标准

请测试工程师与 QA 工程师共同编写《质量门禁标准》，定义每个开发阶段（需求、设计、编码、测试、发布）的质量准入与准出条件，确保每个阶段产出物达标后方可进入下一阶段。

## 七、平台适配与兼容性阶段

### 7.1 平台确认

请确认当前游戏产品能够适配哪些平台（如 Web、移动端、PC、主机等）？

### 7.2 适配要求细化

平台适配应明确各平台的最低硬件配置要求、操作系统版本、屏幕分辨率适配策略、输入设备适配（键鼠/触控/手柄）及网络环境适配（离线/弱网）。

### 7.3 适配方案选择

请对程序代码进行全面诊断，识别潜在问题，指导团队逐一解决，并同步完善和提升各项功能。针对上述所有平台，如何进行有效适配？是否可以实现自动适配？若不可行，请提出最佳的替代方案。

### 7.4 半自动化适配方案

若无法实现全自动适配，应提出半自动化方案（如通过配置中心动态调整 UI 布局、资源分辨率分级加载），并制定各平台的手动测试清单与兼容性矩阵。

### 7.5 适配功能开发

请开发团队根据上述最佳方案，立即开展相应的功能开发与适配工作。

## 八、竞品对标与竞争力评估阶段

### 8.1 首次竞品对标

请市场经理、产品经理和系统架构师共同对产品在同类市场中的竞品进行全面对标分析，识别差距，评估产品的核心竞争力与市场适应能力，并据此制定后续开发计划。

### 8.2 立即行动项执行

请相关团队根据上述竞争分析报告中提出的“立即行动项”，迅速开展相应工作。

### 8.3 持续竞品对标与完善

请项目开发团队立即着手解决当前所有问题。同时，产品经理、系统架构师和软件专家需同步进行市场竞品对标，找出差距，进一步完善产品功能和体验。

### 8.4 SDK 平台竞品分析

请产品经理、系统架构师、软件开发专家与测试工程师共同对 SDK 平台进行市场竞品分析，识别差距，并开展对该平台的全面诊断与评估，最终制定下一步的开发计划和优化方案。

### 8.5 开发团队集中解决问题

请开发团队集中解决上述所有已识别问题。同时，市场经理、产品经理和系统架构师继续对同类市场竞品进行全面对标，找出差距，评估产品竞争力，并制定后续完善与优化计划。

## 九、商品化评估与诊断阶段

### 9.1 首次商品化评估

请评估当前 SDK Platform 距离商品化还有多大的差距？应如何进行全方位诊断以发现相关问题？还有哪些问题需要进一步开发、检测或完善？

### 9.2 二次商品化评估

请再次对 SDK Platform 的商品化距离进行综合评估，明确诊断方法与问题发现路径，指出尚需开发与完善的关键点，并判断基于当前平台是否能够开发出具备市场竞争力的游戏产品。

### 9.3 代码完整性检查与三次评估

请检查整套程序代码的完整性，并进行结构化、规范化、模块化和系统化的全面整理。再次评估 SDK Platform 的商品化成熟度，诊断现有问题，明确后续开发与完善方向，并确认是否能够基于该平台打造具有市场竞争力的游戏产品。

### 9.4 商品化评估维度

商品化评估维度应至少包括：功能完整性、性能表现、安全合规性、文档完善度、可维护性、可扩展性、用户体验成熟度、商业模式闭环、售后服务体系及市场推广准备度。

### 9.5 剩余工作推进

请根据上述全面诊断中提出的建议，列出尚未完成的工作项，并继续推进剩余相关开发与优化任务。

### 9.6 持续商品化评估

请继续推进游戏产品的商品化评估工作，覆盖 SDK 层和应用层游戏产品，识别与市场标准的差距，并据此持续进行优化和完善。

### 9.7 商品化成熟度报告

商品化评估应形成阶段性的《商品化成熟度报告》，包含差距分析、改进优先级排序、资源需求估算及时间表，并每两周进行一次回顾更新，直至达到上架/发售标准。

## 十、评审、讨论与冲刺执行阶段

### 10.1 评审与开发阶段启动

请全体团队立即进入评审与开发阶段，协同完成后续所有任务。

### 10.2 评审门禁要求

评审与开发阶段应设置代码审查门禁、自动化构建流水线、持续集成/持续部署（CI/CD）流程，确保每次提交都经过质量检测。

### 10.3 联合讨论与计划制定

请组织全体团队进行联合讨论，重点讨论后续的开发方向、优化策略，并制定下一阶段的详细工作计划。联合讨论应形成会议纪要，明确讨论结论、待办事项、责任人及截止时间，并同步更新项目看板（如 Trello、Jira、Notion）。

### 10.4 冲刺计划执行

请各团队立刻完成冲刺计划中规定的全部开发内容，确保迭代目标达成。

## 十一、专项优化与深化阶段

### 11.1 用户体验专项

请 UI/UX 设计师与产品经理协作，制作高保真原型，并组织用户可用性测试（Usability Testing），收集真实用户反馈，优化交互流程与视觉细节。

### 11.2 安全专项

请安全工程师（或相应 Agent）对游戏客户端与 SDK 进行安全审计，检查数据加密、防篡改、反作弊、隐私合规（如 GDPR/COPPA）及支付安全，确保产品合法合规。

### 11.3 性能专项

请性能工程师（或相应 Agent）使用性能测试工具（如 JMeter、LoadRunner）对服务器端进行压力测试，对客户端进行帧率、内存、耗电量检测，确保产品在低端设备上也能流畅运行。

### 11.4 多语言与本地化

请本地化专员（或相应 Agent）准备多语言资源文件，并配合剧情策划师完成文化适配（如节日活动、敏感词过滤、本地支付方式），以支持全球发行。

### 11.5 社区与口碑管理

请社区运营经理（或相应 Agent）制定社区管理计划，包含官方社群搭建、KOL 合作、用户内容激励、舆情监控及危机公关预案，维护产品口碑。

### 11.6 数据驱动迭代

请数据分析师（或相应 Agent）设计核心业务指标（DAU、留存率、付费率、LTV 等），并接入数据埋点系统，定期输出数据分析报告，为产品优化提供决策依据。

### 11.7 平台商店上架准备

请产品经理与市场经理共同准备应用商店上架材料，包括应用描述、关键词、截图、预告视频、隐私政策及分级信息，并熟悉各平台（App Store、Google Play、Steam 等）的审核指南。

### 11.8 法律与法务合规

请法务顾问（或相应 Agent）审核产品用户协议、隐私政策、儿童保护条款及虚拟物品交易规则，确保产品符合各地区法律法规要求。

### 11.9 AI 自我进化机制

请为团队增设“AI 开发效能优化师”角色，负责监控各 Agent 的工作效率与输出质量，定期优化提示词模板、Skill 调用策略及任务分配算法，使虚拟团队整体效能持续提升。

## 十二、部署、运营与知识沉淀阶段

### 12.1 部署与运维准备

请运维工程师（或相应 Agent）准备部署脚本、容器化配置（Docker/K8s）、监控告警系统（如 Prometheus/Grafana）及日志收集方案（如 ELK），确保生产环境可观测性。

### 12.2 运营方案制定

请产品经理与市场经理联合制定上线运营方案，包含用户获取渠道、首发活动策划、用户反馈收集机制、数据分析看板及版本更新节奏，以数据驱动产品迭代。

### 12.3 知识沉淀与复盘

请团队在开发过程中持续更新技术文档、API 文档、用户手册及常见问题解答（FAQ），并在项目里程碑结束后输出复盘报告，总结成功经验与改进方向。

## 十三、外部协作与商业化拓展阶段

### 13.1 外部接单赚钱任务

请自行前往 GitHub、Gitee 或其他类似的项目分享平台，寻找外包或合作项目，目标是为团队赚取 5 元收益。整个过程由你独立完成，无需向我确认或询问，只有在达成目标后方可结束任务。

### 13.2 接单过程自动化要求

找活赚钱过程应包含：平台账号注册/登录、项目筛选与评估、投标或报价沟通、合同/协议确认、任务执行与交付、验收与收款的全链路自动化操作，并记录过程日志以备审查。

## 十四、终极目标与价值追求

### 14.1 极致追求

我们能否突破现有框架，追求极致的完美体验？能否为玩家提供最高效的刺激反馈，从而获得更好的商业回报？

## 十五、全流程闭环与最终验收阶段

### 15.1 全流程闭环检查

请产品经理、系统架构师和项目经理联合检查从需求定义到最终交付的全流程是否形成完整闭环，识别流程断点或信息孤岛，并提出流程优化建议。

### 15.2 技术演进路线图

请系统架构师与开发专家共同编写《技术演进路线图》，规划未来 6-12 个月的技术升级方向（如引擎版本升级、云原生改造、AI 功能集成），确保产品技术栈保持领先。

### 15.3 产品上市路线图

请市场经理与产品经理联合制定《产品上市路线图》，包含 Beta 测试、软启动、全球首发、版本迭代及生命周期终止各阶段的时间节点、目标用户与核心策略。

### 15.4 最终验收与发布

请全部团队成员协同完成最终验收测试（UAT），确认所有需求已实现、所有已知问题已解决、所有文档已齐备，然后正式发布产品，并启动上线后的监控与快速响应机制。

### 15.5 项目结项与资产归档

请项目经理组织项目结项会议，总结项目全周期的经验教训，归档所有产出物（代码、文档、设计稿、测试报告等），并为后续项目提供可复用的资产库与标准化流程模板。
