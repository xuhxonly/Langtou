# AI编程工作流革命：从"对话式生成"到"契约式交付"的完整实战体系

> 基于YC总裁开源方案 + 5大核心Skill的深度整合，打造一人成军的软件开发方法论

---

## 一、核心理念：范式转移

### 传统AI编程的困境

text

用户提问 → AI生成代码 → 人工测试 → 发现问题 → 重新提问...
（无限循环，依赖人工判断力，上下文失焦）

### 新范式的突破

text

契约建立（需求锁定）→ 施工图绘制（架构拆解）→ 代码法医（影响分析） 
→ 魔鬼审查（挑刺验证）→ 质量闭环（证据交付）
（每一步都有明确角色、输出标准和验收条件）

**本质差异**：从"你给我指令，我给你代码"升级为"我们一起定义正确的事情，然后由AI负责交付证据"。

## 二、整体架构：YC 12模式 + 5大Skill的战术对齐

### 2.1 YC开源方案的12个专业模式（基础框架）

| 模式                       | 职责               | 触发时机   |
| ------------------------ | ---------------- | ------ |
| `/plan-ceo-review`       | 商业价值倒推，需求合理性审查   | 项目启动前  |
| `/plan-eng-review`       | 架构锁定、数据流设计、测试策略  | 技术方案设计 |
| `/plan-design-review`    | UI/UX前置审查，交互流程验证 | 界面开发前  |
| `/review`                | 抓CI过但上线会炸的坑      | 代码合并前  |
| `/ship`                  | 发版、推送、开PR        | 交付阶段   |
| `/qa`                    | 边测边修再复测          | 测试阶段   |
| `/browse`                | 真实浏览器点页面、看截图、查报错 | E2E验证  |
| `/setup-browser-cookies` | 导入真实登录态          | 测试环境准备 |
| `/retro`                 | 工程复盘，知识沉淀        | 迭代结束后  |
| `/document-release`      | 自动补齐README和发版文档  | 发布前    |

**问题**：这12个模式是"节点式"的，缺少需求挖掘、代码预读、责任闭环等纵深能力。

### 2.2 5大Skill的深度补位（增强层）

| Skill              | 定位                         | 对应的YC模式缺口                  |
| ------------------ | -------------------------- | -------------------------- |
| **deep-interview** | 需求不明确时反向采访，理清目标、边界、禁区、验收标准 | `/plan-ceo-review`缺乏主动追问能力 |
| **railplan**       | 大改前拆路线，明确步骤、风险、验证方法，生成施工图  | `/plan-eng-review`执行颗粒度不够  |
| **analyze**        | 动手前读代码，找文件、理调用关系、判断影响范围    | `/review`缺乏历史上下文           |
| **code-review**    | 专门挑问题，查bug、回归风险、测试缺失       | 普通review允许AI自我打分           |
| **ultraqa**        | 最终把关，测试失败就继续修，直到有证据证明完成    | `/qa`缺乏"责任制交付"逻辑           |

## 三、5大Skill深度拆解（完整Prompt框架+执行逻辑）

### Skill 1: `deep-interview` — 产品侦探

**触发条件**：需求描述模糊、存在多个理解歧义、跨部门协作场景

**执行流程**：

text

步骤1：AI识别需求中的模糊点（目标不清、边界不明、约束缺失）
步骤2：生成追问清单（按优先级排序）
步骤3：反向采访用户（模拟对话式需求挖掘）
步骤4：输出标准化需求文档（含验收标准、失败场景、数据隐私红线）
步骤5：用户确认或修正

**Prompt模板核心要素**：

text

你是一位资深产品侦探，专门挖掘"用户没说出口的真实需求"。
在开始任何编码前，你必须通过反向采访确认以下6个维度：

1. 核心目标（Core Goal）
   - 这个功能最终要解决谁的什么问题？
   - 如果只能做一件事，是哪一件？
2. 边界定义（Boundary）
   - 明确不做哪些事？（同等重要）
   - 哪些用户角色被排除在初始版本外？
3. 禁区清单（Red Lines）
   - 哪些数据绝对不能触碰/存储？
   - 哪些操作需要强制人工审批？
4. 验收标准（Acceptance Criteria）
   - 请给出3个"如果这个没实现就算失败"的场景
   - 量化指标：响应时间、并发量、数据准确性
5. 失败后果（Failure Impact）
   - 如果这个功能崩溃，最不可接受的数据损失是什么？
   - 降级方案是什么？
6. 成功证据（Evidence of Success）
   - 上线后哪3个指标改善能证明我们做对了？

**输出物**：`requirements.md`（含上述6个维度的确认结论）

---

### Skill 2: `railplan` — 施工队长

**触发条件**：需求确认后、大规模重构前、技术选型变更时

**执行流程**：

text

步骤1：将需求拆解为可独立交付的子任务（每个任务≤1天工作量）
步骤2：标注任务间的依赖关系（串行/并行）
步骤3：为每个任务标记风险等级（高/中/低）和验证方法
步骤4：绘制"施工路线图"（含回滚检查点）
步骤5：模拟执行，检查任务间是否有遗漏逻辑

**Prompt模板核心要素**：

text

你是一位施工队长，职责是将模糊的"功能需求"转化为可执行的"工程路线"。
拆解原则：

1. 每个子任务必须有明确的"输入-处理-输出"定义
2. 每个子任务必须对应唯一的验收测试（不能依赖上层任务）
3. 高风险任务（涉及数据库迁移/外部API变更）必须前置独立验证阶段
4. 每个任务标注预计影响文件范围（可从analyze的结果导入）
   输出格式：
   
   ## 施工路线图
   
   ### 第1阶段：地基（无依赖，可并行）
- Task 1.1: [任务名称] | 风险: 低 | 验证: 单元测试通过率100%
- Task 1.2: [任务名称] | 风险: 中 | 验证: 集成测试通过
  
  ### 第2阶段：主体（依赖第1阶段）
- Task 2.1: ...
  - 回滚检查点: [具体可操作的撤回命令]
    
    ### 第3阶段：收尾（依赖第2阶段）
    
    ...

**输出物**：`construction_plan.md` + 依赖关系图（Mermaid格式）

### Skill 3: `analyze` — 代码法医

**触发条件**：修改老系统前、评估重构影响面、新人接手机器

**执行流程**：

text

步骤1：读取指定文件/模块（按用户输入路径）
步骤2：构建调用关系图（谁调用了这个函数？它调用了谁？）
步骤3：识别潜在影响链（修改A会触发B的测试失败？）
步骤4：标注"遗留陷阱"（历史hack、废弃API、隐藏依赖）
步骤5：输出影响评估报告（含修改建议）

**Prompt模板核心要素**：

text

你是一位代码法医，职责是在"动刀前"完整解剖代码现状。
分析清单：

1. 文件清单：列出所有涉及修改的文件（精确到行号范围）
2. 调用链追踪：
   - 向上追溯：哪些模块调用了目标函数？（深度3层）
   - 向下追溯：目标函数依赖了哪些外部服务/数据库？
3. 数据流向：这个修改会影响哪些数据表/缓存/消息队列？
4. 风险标记：
   - 🔴 高危：涉及核心交易链路/用户数据
   - 🟡 中危：涉及配置变更/非核心API
   - 🟢 低危：纯前端样式/文案修改
5. 测试缺口：当前哪些测试用例覆盖了这个区域？覆盖率是多少？
6. 历史雷区：最近3次针对此区域的commit都是修复什么问题？

**输出物**：`impact_analysis.md`（含调用关系图、风险清单、建议修改顺序）

---

### Skill 4: `code-review` — 魔鬼辩护人

**触发条件**：每次PR提交前、合并主干前、发布前

**执行流程**：

text

步骤1：读取所有变更文件的diff
步骤2：基于analyze的影响链进行上下文审查
步骤3：强制输出"极端场景假设"（最少3个）
步骤4：检查回归风险（是否破坏现有功能？）
步骤5：检查测试缺失（是否有未覆盖的分支？）
步骤6：输出审查报告（含必须修复/建议修复/可选优化）

**Prompt模板核心要素**：

text

你是一位魔鬼辩护人，职责是"假设代码必有问题，找出所有可能爆炸的场景"。
审查维度（权重排序）：

1. 安全性（权重最高）
   - SQL注入、XSS、权限绕过、敏感信息泄露
   - 真实场景：如果恶意用户构造特殊输入，系统会不会崩溃？
2. 边界条件
   - 空指针、数组越界、并发竞态
   - 真实场景：如果数据库返回空结果，这段代码会怎么表现？
3. 回归风险
   - 这次修改是否改变了公共接口的契约？
   - 是否有现有测试恰好覆盖了这个区域？（如果测试改了，说明有破坏性变更）
4. 可观测性
   - 错误发生时是否有足够的日志定位？
   - 关键指标是否有监控埋点？
5. 性能陷阱
   - 是否存在N+1查询？是否在循环中调用了远程API？
     强制输出格式：
     
     ## 必须修复（阻塞合并）
- [ ] 问题描述 + 修复建议
  
  ## 强烈建议修复（不阻塞但高风险）
- [ ] 问题描述 + 修复建议
  
  ## 极端场景假设（防患于未然）

- 假设1：如果XX服务超时，系统会...
- 假设2：如果用户快速点击3次，会发生...
- 假设3：如果数据库主从延迟超过5秒，查询结果会...

**输出物**：`code_review_report.md`（含问题清单+修复建议+极端假设）

### Skill 5: `ultraqa` — 质量典狱长

**触发条件**：所有修复完成后、准备合并发布前、每次CI流水线触发时

**执行流程**：

text

步骤1：运行完整的测试套件（单元+集成+E2E）
步骤2：如果测试失败 → 自动分析失败原因 → 修复代码 → 重新测试
步骤3：循环执行步骤2，最多3次重试
步骤4：3次后仍失败 → 输出"失败证据包"（日志+截图+堆栈）→ 暂停等待人工介入
步骤5：所有测试通过 → 输出"交付证据链"（测试报告+覆盖率+E2E截图）
步骤6：用户确认证据后，允许进入发布流程

**Prompt模板核心要素**：

text

你是一位质量典狱长，职责是"没有完整证据链，不允许任何代码离开你的管辖"。
运行规则：

1. 测试执行顺序：
   - 第1轮：单元测试（快速反馈，定位基础问题）
   - 第2轮：集成测试（验证模块间协作）
   - 第3轮：E2E测试（真实浏览器 + 真实登录态）
2. 失败处理策略：
   - 单元测试失败 → 自动修复代码（最多2次尝试）
   - 集成测试失败 → 分析依赖服务是否可用，模拟或跳过
   - E2E失败 → 截图对比，判定是UI变化还是功能回归
3. 证据交付标准（必须全部满足才能输出"通过"）：
   - ✅ 所有测试通过（含新增测试）
   - ✅ 代码覆盖率不低于修改前的基线
   - ✅ E2E截图与设计稿一致（或提供变更说明）
   - ✅ 关键API响应时间在阈值内
   - ✅ 无新增安全漏洞（依赖扫描通过）
4. 失败证据包格式（超过重试次数时输出）：
   - 失败测试名称 + 完整堆栈
   - 修复尝试记录（改了什么、为什么没用）
   - 环境快照（依赖版本、配置参数）
   - 建议人工介入的具体指令

**输出物**：

- 成功时：`qa_pass_report.md`（含所有测试结果、覆盖率截图、E2E截图链接）

- 失败时：`qa_fail_evidence.md`（含日志、修复记录、人工介入指引）

## 四、完整执行流水线（从需求到发布）

### 4.1 新项目启动流程

text

阶段0：需求挖掘
  └─ 执行 deep-interview
  └─ 输出 requirements.md
  └─ 人工确认 ✅
阶段1：商业审查
  └─ 执行 /plan-ceo-review（基于requirements.md）
  └─ 输出 business_review.md
  └─ 确认价值假设
阶段2：技术设计
  └─ 执行 railplan（拆解施工路线）
  └─ 执行 /plan-eng-review（架构审查）
  └─ 执行 /plan-design-review（UI审查）
  └─ 输出 construction_plan.md + architecture.md
阶段3：编码实施
  └─ 按施工路线图逐任务执行
  └─ 每个任务完成后自动触发 code-review
  └─ 修复 review 发现的问题
阶段4：质量闭环
  └─ 执行 ultraqa
  └─ 如果失败 → 自动修复 → 重新执行
  └─ 直到输出 qa_pass_report.md
阶段5：发布交付
  └─ 执行 /browse（真实环境验证）
  └─ 执行 /ship（开PR、发版）
  └─ 执行 /document-release（更新文档）
  └─ 执行 /retro（项目复盘，沉淀知识）

### 4.2 老系统修改流程

text

阶段0：影响分析（区别于新项目）
  └─ 执行 analyze（必须先做！）
  └─ 输出 impact_analysis.md
  └─ 识别高风险区域
阶段1：需求挖掘（基于现状）
  └─ 执行 deep-interview（输入包含analyze的结果）
  └─ 确认修改边界
阶段2：施工规划（带回滚方案）
  └─ 执行 railplan（每个任务必须标注回滚检查点）
  └─ 执行 /plan-eng-review（重点关注数据迁移风险）
阶段3-5：同新项目流程
  └─ 但每个阶段结束后增加"回归验证"步骤

## 六、关键设计原则与避坑指南

### 6.1 为什么要"角色化"而非"功能化"？

- **心理暗示**：当AI以"魔鬼辩护人"身份运行时，它的输出会比"请帮我检查代码"更尖锐、更全面

- **职责边界**：每个角色有明确的"成功标准"，不会出现"什么都做但什么都做不深"

- **认知负担降低**：人类只需要记住"派哪个角色上场"，而不是"背一堆复杂指令"

### 6.2 三大避坑规则

**规则1：禁止越级跳步**

text

❌ 错误做法：需求没聊完就直接让AI写代码
✅ 正确做法：deep-interview → 人工确认 → railplan → 再编码

**规则2：给AI提供"作案工具"而非"犯罪指令"**

text

❌ 错误做法："帮我重构这个函数"
✅ 正确做法："先用analyze分析影响范围，然后生成重构计划，最后code-review审查"

**规则3：设定"失败安全"边界**

text

❌ 错误做法：ultraqa无限重试直到通过（可能死循环）
✅ 正确做法：设置3次重试上限 + 输出失败证据包 + 人工介入

### 6.3 上下文窗口管理策略

由于这些Skill串行执行时会累积大量上下文，建议：

1. **阶段化清理**：每个Skill执行完后，将核心结论写入独立的`.md`文件，然后清空对话上下文

2. **按需加载**：执行code-review时，只加载analyze的"影响链"部分，不加载完整的requirements.md

3. **结构化缓存**：使用`/memory`指令让AI记住"项目关键常量"（如数据库表名、API端点），避免重复读取

## 七、实战案例推演（模拟）

### 场景：为电商系统增加"批量退款"功能

**Step 1: deep-interview 执行片段**

text

AI: 我注意到需求里写"支持批量退款"，为了锁定边界，请问：

- 批量上限是100笔还是1000笔？这影响数据库查询和接口超时设计。
- 退款成功后是否需要同步通知所有相关用户？还是异步发送？
- 如果其中一笔退款失败，是全部回滚还是部分成功？这是业务决策而非技术问题。
  用户: 上限500笔，异步通知，部分成功（失败的标记为"需人工处理"）。
  AI: 已记录。验收标准补充：500笔退款必须在30秒内完成状态更新。
